# 赤月孤城 PWA 游戏项目交接文档（最终交接版）

本文档用于让新的 AI / 开发者快速接手本项目。请先完整阅读本文档，再改代码。

项目当前不是成品，而是一个已经能运行、能在手机上玩、具备基础战斗和掉装循环的早期 MVP。后续重点不是重写，而是在现有基础上逐步把它做成耐玩的单机类传奇游戏。

---

## 1. 项目概述

项目名：**赤月孤城**

目标：开发一个单机耐玩的类传奇 / 类 Mir 刷怪 RPG。

核心要求：

- 单机运行，不依赖服务器。
- PC 浏览器可玩。
- iOS Safari 可玩。
- 可作为 PWA 添加到 iOS 主屏幕。
- 本地自动存档。
- 极简终端符号风美术。
- 保留一定物理反馈：击退、震屏、粒子、碰撞。
- 最终目标是耐玩，而不是只做 Demo。

当前状态：

- 已经有完整项目结构。
- 已经能运行和构建。
- 已经有基础刷怪、升级、掉装、背包、装备、商店、Boss、存档。
- 已经做了移动端竖屏适配。
- 已经做了攻击前摇、怪物攻击前摇、攻击轨迹、实体碰撞。
- 仍然是 MVP，很多系统需要继续打磨。

## 2. 技术栈

当前技术：

- Vite
- TypeScript
- HTML5 Canvas
- PWA manifest
- service worker
- localStorage 存档

不要轻易更换技术栈。选择 Web PWA 的原因：

- iOS 可以直接用 Safari 打开。
- 可以添加到主屏幕。
- 不需要 Xcode、签名、App Store。
- PC / iOS / 安卓浏览器都能跑。
- 开发和调试成本低。

## 3. 运行方式

安装依赖：

```bash
npm install
```

启动开发服务器：

```bash
npm run dev
```

本机访问：

```text
http://localhost:5173
```

构建检查：

```bash
npm run build
```

手机测试：

1. 电脑和手机连接同一个 Wi-Fi。
2. 获取电脑局域网 IP。
3. 手机 Safari 打开：

```text
http://电脑局域网IP:5173
```

PWA 测试：

- 在 iOS Safari 里选择“添加到主屏幕”。
- 如果 manifest 或显示模式改过，旧图标可能不更新，需要删除主屏幕旧图标后重新添加。

## 5. 核心文件说明

### 5.1 `src/main.ts`

入口文件。

职责：

- 引入样式。
- 获取 `#game` canvas。
- 创建 `Game` 实例。
- 启动游戏循环。
- 注册 service worker。
- 暴露 `window.game` 方便调试。

### 5.2 `src/game/Game.ts`

最核心文件。

职责：

- 创建初始游戏状态。
- 主循环 update。
- 玩家移动。
- 怪物 AI。
- 普攻 / 技能 / 怪物攻击。
- 攻击前摇、命中、后摇。
- 实体碰撞。
- 击退、震屏、粒子。
- 掉落拾取。
- 背包使用 / 装备。
- 商店买药 / 出售。
- 死亡和复活。
- 本地存档。
- 手机端背包点击交互。

这个文件偏大，但不要急着大重构。优先保证玩法稳定。

### 5.3 `src/game/render.ts`

Canvas 渲染文件。

职责：

- 画世界网格。
- 画地图区域。
- 画玩家、怪物、掉落。
- 画攻击轨迹 `SlashEffect`。
- 画粒子。
- 画 HUD。
- 画背包。
- 画商店。
- 画死亡界面。
- 画横屏 / 竖屏触摸控件。

美术风格为极简终端符号风：

- 玩家：`@`
- 稻草人：`#`
- 半兽人：`&`
- 骷髅战士：`%`
- 沃玛卫士：`W`
- 赤月恶魔：`Ω`
- 金币：`$`
- 装备：`◇`
- 药水：`+`

### 5.4 `src/game/input.ts`

输入系统。

支持：

- 键盘移动。
- 键盘技能和背包操作。
- 触摸摇杆。
- 触摸按钮。
- 横屏 / 竖屏按钮坐标。
- 点击坐标换算，避免 canvas CSS 尺寸和实际尺寸不一致导致错位。

重要：如果改 `render.ts` 中按钮位置，一定要同步改 `input.ts` 的命中检测。

重要类型：

## 11. 代码质量要求

后续 AI 代码质量可能不如当前，所以请严格遵守：

### 11.1 不要大重构

不要一次性重写整个项目。

每次只做一个明确任务：

- 只改背包 UI。
- 只改战斗手感。
- 只改怪物 AI。
- 只改地图。

### 11.2 每次改完必须构建

```bash
npm run build
```

构建不过，不要继续加功能。

### 11.3 注意旧存档兼容

localStorage 中可能已有旧玩家数据。

新增字段要补默认值，例如：

```ts
player.facing ??= { x: 1, y: 0 };
player.action = null;
```

### 11.4 Canvas UI 要同步渲染和点击区域

`render.ts` 负责画按钮。

`input.ts` / `Game.ts` 负责点击命中。

如果只改了显示位置，没改点击命中，会出现“看得到但点不到”。

### 11.5 横屏和竖屏都要考虑

已有判断：

```ts
function isPortrait(w: number, h: number): boolean {
  return h > w * 1.12;
}
```

新增 UI 时必须考虑竖屏布局。

### 11.6 不要过早抽象

这是游戏项目，当前更重要的是体验。不要为了架构漂亮做大拆分，除非当前任务真的需要。

## 13. 设计意图和小巧思

后续接手时要尽量保留这些设计方向，不要把项目改成普通按钮网页或普通像素 demo。

### 13.1 视觉方向

当前美术不是“没素材所以占位”，而是有意做成 **极简终端符号风**：

- 黑绿色终端底色。
- 线框 UI。
- 符号表示实体。
- 发光文字和符号粒子制造氛围。
- 怪物、掉落、技能都优先用字符表达。

这个方向的好处：

- 不依赖外部素材。
- 很适合 AI 快速扩展。
- 文件体积小。
- PWA 加载快。
- iOS 性能压力低。
- 有辨识度，不像普通粗糙占位图。

后续不要随便引入大图素材。如果要升级美术，优先升级“字符动画、线框特效、符号组合、颜色层次”，而不是改成贴图游戏。

### 13.2 战斗方向

战斗要逐步靠近“有节奏的刷怪 ARPG”，不是纯点击扣血。

已经确立的原则：

- 玩家攻击有前摇和后摇。
- 怪物攻击也有前摇，玩家应该能看懂危险。
- 命中要有斩击线、粒子、击退、震屏。
- 怪物不能堆成一个点，要有身体体积。
- Boss 应该更重、更慢、更有压迫感。

后续加技能时，也应遵守这个方向：

- 技能要有可见轨迹。
- 技能要有不同用途，而不是只改倍率。
- 怪物强招要有警告线或警告符号。

### 13.3 手机优先

虽然 PC 也能玩，但用户明确希望 iOS 能运行，所以后续 UI 设计要手机优先。

关键原则：

- 竖屏不能遮挡主要战斗区域。
- 右手负责攻击和技能。
- 左手负责移动。
- 背包、商店、装备必须能纯触摸完成。
- 不要依赖 Enter / 键盘逻辑。
- 新按钮渲染在哪里，点击命中就必须在哪里。

### 13.4 单机耐玩方向

最终体验应该是：

```text
刷怪 -> 掉装 -> 对比装备 -> 变强 -> 进入高级区域 -> 挑战 Boss -> 解锁更强目标
```

短期不要做复杂剧情。更应该先强化：

- 装备选择
- Boss 掉落
- 地图推进
- 简单任务目标
- 强化系统

### 13.5 不要丢掉当前手感基础

当前已有的前摇、击退、震屏、碰撞、粒子虽然还粗糙，但它们是游戏从 demo 走向可玩的基础。后续改战斗时不要退回“按下立即扣血”的模式。

## 15. 给接手 AI 的能力增强提示词

下面这段不是给用户看的需求描述，而是给接手 AI 用来校准工作方式的“能力增强提示”。建议在新 AI 开始写代码前，把这段和本文档一起给它。

```text
你现在接手的是一个正在从 MVP 走向可玩成品的单机 Canvas RPG。请把自己当作“游戏玩法工程师 + 移动端交互设计师 + TypeScript 维护者”，而不是只会补代码的助手。

你的工作目标不是堆功能，而是持续提高游戏的可玩性、手感和可维护性。每次改动前，先判断这个改动会不会让玩家更容易理解、更愿意继续玩、更少误触、更有反馈。如果答案不明确，就收窄范围，只做最确定的改进。

请遵守以下工作方式：

1. 先理解现有系统，不要急着重写
   - 这个项目已经有一套可运行的循环：移动、战斗、掉落、装备、背包、商店、存档、PWA、竖屏触摸。
   - 不要因为代码不完美就大重构。
   - 优先沿用现有类型和函数，把体验补完整。

2. 游戏体验优先于架构洁癖
   - 玩家能不能看懂发生了什么，比代码分层是否完美更重要。
   - 攻击必须有反馈，受击必须有反馈，按钮必须能点准。
   - UI 需要直接服务移动端玩家，而不是服务键盘用户。

3. 保留并强化“极简终端符号风”
   - 当前不是素材缺失，而是刻意选择的视觉方向。
   - 优先用字符、线框、发光、粒子、轨迹、颜色表达内容。
   - 不要引入大贴图，不要改成普通网页按钮风格。
   - 新 UI 也应该像终端面板，而不是普通表单。

4. 手机端是第一体验
   - 所有重要操作必须能纯触摸完成。
   - 不要让 Enter、键盘方向键、悬浮 hover 成为必要操作。
   - 竖屏下 HUD、背包、商店、按钮不能遮挡战斗核心区域。
   - render.ts 里画了什么，input.ts / Game.ts 就必须能点到什么。

5. 战斗不能退回“瞬间扣血”
   - 现有战斗已经有 windup / strike / recovery。
   - 后续所有玩家攻击、怪物攻击、Boss 技能都应该遵守“预兆 -> 命中 -> 恢复”的节奏。
   - 玩家应该能通过观察符号、线条、颜色判断危险。
   - 命中要有击退、粒子、斩击线或其他明确反馈。

6. 做 Canvas UI 时要有 hit-test 意识
   - Canvas 上没有真实 DOM 按钮。
   - 每个看起来能点的区域，都必须有对应命中判断。
   - 横屏和竖屏的坐标要分开考虑。
   - 如果改了按钮位置，必须同步改点击区域。
   - 如果 UI 面板打开，应该避免误触发攻击或摇杆。

7. 新增状态要兼容旧存档
   - 存档在 localStorage。
   - 玩家可能已经有旧数据。
   - 给 Player、Monster、GameState 新增字段时，要在 createState 或读取后补默认值。
   - 不要因为新增字段导致旧存档白屏。

8. 每次只推进一个明确目标
   - 例如：只重做背包触摸交互；只打磨怪物攻击警告；只做装备对比。
   - 不要同时改战斗、背包、地图、存档、数值。
   - 小步稳定迭代比一次性大改更适合这个项目。

9. 写代码时优先可读和直接
   - TypeScript 类型要明确。
   - 不要为了炫技引入复杂框架或抽象。
   - 重复两三处可以接受，等玩法稳定后再抽象。
   - 不要留下半成品入口、无用状态、死代码。

10. 修改后必须自检
   - 至少运行 npm run build。
   - 检查 TypeScript 错误。
   - 思考 PC 横屏、手机竖屏、PWA 主屏幕三种场景是否会坏。
   - 特别检查点击是否错位、UI 是否超出屏幕、旧存档是否兼容。

如果你要继续当前最高优先级任务，请先重做手机背包/装备系统：做触摸网格、物品详情、装备属性对比、装备/使用/出售/丢弃/关闭按钮，并保证横屏和竖屏都可操作。不要改变项目整体风格，不要重写技术栈，不要破坏现有战斗状态机。
```
